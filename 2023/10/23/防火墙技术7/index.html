UDP协议问题

UDP协议提供应用程序（通过UDP端口标识）用来发送和接收数据报的一种机制，它没有TCP连接的功能。

UDP协议本身所做的工作很少，UDP不负责重新发送丢失的或出错的数据报、不对接收到的无序数据报重新排序、不消除重复的数据报、不确认数据报已经收到、也不负责建立或终止连接。传输层对UDP的传输几乎没有任何控制功能，使用UDP的应用程序要承担可靠性方面的全部工作。

状态检测防火墙针对UDP的处理

防火墙收到数据包，确认传输层协议：TCP或者UDP

数据包是UDP，直接根据四元组查找状态表，如果能够找到，直接转发；如果找不到，查找规则表

如果规则表允许转发，在状态表中增加一条记录，记录该UDP数据的四元组

但是UDP没有连接释放的过程，如何删除状态表中的记录？（超时判断）

防火墙状态表的基本要素

协议类型（TCP或者UDP）

四元组（2个IP地址、2个端口号）

超时判断：记录的生成时间，当生成时间超时（30钟无数据传输），自动删除该记录

协议类型 IP地址1 IP地址2 端口1 端口2 生成时间

多通道协议问题

单通道协议：从会话建立到删除的全过程中，只有一个通道参与数据交互，如 HTTP。

多通道协议：包含一个控制通道和若干其它控制或数据通道，即控制信息的交互和数据的传送是在不同的通道上完成的，如 FTP。

多通道协议工作过程

FTP 连接的建立过程如下：假设 FTP client（10.1.1.9） 以 1333 端口向 FTP server（10.1.1.1） 的 21 端口发起 FTP 控制通道的连接，通过协商决定由 FTP server 的 20 端口向 FTP Client 的 1600 端口发起数据通道的连接，数据传输超时或结束后连接删除。

防火墙对多通道协议的处理

防火墙检查从出接口上向外发送的 IP 报文，确认为基于 TCP 的 FTP 报文；（TCP端口：21）

检查端口号确认连接为控制连接，建立状态表；

检查 FTP 控制连接报文，解析 FTP 指令（应用层协议），如果包含数据通道建立指令，则创建数据连接的临时状态表，或者建立状态表；

FTP 连接删除时，状态表及临时状态表随之删除。

1、SQL注入漏洞

漏洞简介

WEB程序将客户端输入当作SQL语句执行

漏洞成因

对用户输入中包含的SQL关键字过滤不严

2、Cookie欺骗漏洞

  • 浏览器与WEB服务之间通过多个不同会话连接进行通信

  • 用户登录WEB服务成功之后，WEB服务给浏览器分配一个Cookie，该Cookie往往包含用户身份信息，例如通过base64（用户名）转码得到Cookie

  • 此后该浏览器访问该WEB服务的所有URL中都带有该Cookie，WEB服务对该Cookie进行身份验证

1、登录界面，生成Cookie

2、登录成功后，后续界面只要验证Cookie

3、跨站脚本漏洞

  • 跨站脚本是当今网络上最普遍的漏洞之一， 可能会导致各种后果，例如帐户泄露、帐户删除、权限提升、恶意软件感染、窃取Cookie等。

  • 跨站脚本攻击是指恶意攻击者往Web页面里插入恶意脚本代码，当用户浏览该页之时，嵌入其中Web里面的脚本代码会被执行，从而达到恶意攻击用户的目的。

  • 存储型跨站脚本攻击：攻击代码存储在服务器中的，如在个人信息或发表文章等地方，加入代码，如果没有过滤或过滤不严，那么这些代码将储存到服务器中，每当有用户访问该页面的时候都会触发代码执行。

WAF主要技术

WAF的基本功能：

Web入侵检测，根据入侵的特征预设规则。其原理是每一个会话都要经过一系列的检测，每一项检测都由一个或多个规则组成，如果检测没通过，请求就会被认为非法而被拒绝；

基于规则的WAF很容易构建并且能有效地防范已知的安全问题，但这必须要先确定每一个威胁的特点，进而构建出一个庞大的规则库来支持检测。而且这个规则库要不断维护更新；

WAF的高级功能：

网页自学习技术，通过检测一段时间的用户访问，WAF记录了常用网页的访问模式，如一个网页中有几个输入点，输入的是什么类型的内容，通常情况的长度是多少等。

学习完毕后，定义出一个网页的正常使用行为模型，当有用户行为突破了这个模型，比如一般的帐号输入不应该有特殊字符，WAF就会根据预先定义的方式进行预警或予以阻断；再如密码长度一般不超过20位，在SQL注入时加入代码会很长，同样突破了网页访问的行为模型。
